### 2. Data Integrity Issues

- Incorrect Handling of Multi-Polygons/Exclaves:
  Does `getGeometryForRelation`, `geojson.features.find(...)` select the first Polygon it finds? OSM sometimes returns duplicate entries for the same region, some of which are incorrect. To deal with this, I was thinking of something like this (I'm not sure if the existing code follows this accurately):

  A. Run an initial filter that drops osm entries that have an 'ine:\*' property (OSM only includes 'ine:ccaa', 'ine:provincia', and 'ine:municipio' properties) AND whose 'ine:\*' property value is incorrect (so, for example, if a row in the madrid file has an 'ine:municipio' value that doesn't match any of madrid's municipality ids.) AND whose name (cand.rel.tags) has a string similariry that is too low compared to any of it's admin_level peers (the names in the tables). This would drop the dirtiest and least relevant entries from being processed in the following steps.

  B. Run a second filter, but now looping through all the geo location table rows (communities, provinces, and municipalities) corresponding to the given osm.pbf file being processed. The purpose of this second pass is to try to drop incorrect duplicates, keeping at least one osm entry for every geo location corresponding to the file's CCAA (so one osm entry for the given community and one for each of its provinces and municipalities.) Remember, while processing a given community file, we do not want to consider any possible matches for _other_ communities and their children.

  Now, for every geo location, we can check all the osm entries that potentially correspond to it, either by 'ine:\*' value or by string similarity (those we didn't drop in step 1). If there's only one potential match, we can keep it for the last filter and loop to the next location. If there are none, we should console log it. And if there are multiple, we can run a series of checks for incorrect duplicates: starting with 'ine:\*' codes, we will see which of the potential entries have the correct 'ine:\*' value for the location. If only one does, we keep it and continue. If none or multiple do, then we check the similarity scores of the potential entries with the location. If one surpases the others, we keep it. If they all have the same osm_name, then we pass app of these onto the 3rd polygon-based filter.

  C. For the 3rd pass, we will take the surviving set of duplicate and sole entries for every location, and we will see if their polygons fall into their parent's polygon. If the entry doesn't have a parent, because it's a ccaa, or if its parent doesn't have a polygon, throw an error. If the parent has a polygon, then check if the survinging entries' coordinates line up. If only one of the surviving entries has coordinates that fall within the parent's polygon, great; we can assign it. If none do, we drop it and console log it. If multiple do, we can take the union of the survining polygons and assign the union as the location's polygon. In this last union case, if there are other values in the entries nested JSONs, like for the 'population' properties, we can assign the mean value rounded to the nearest integer.
